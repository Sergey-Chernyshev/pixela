# Frontend — Angular Decisions

> Обоснования архитектуры дашборда. Claude Code: следуй этим решениям. Это самая UX-чувствительная часть продукта.

## Standalone components + signals (без NgModules)

**Почему.** Современный Angular (standalone API + signals) убирает бойлерплейт NgModules и даёт реактивное
состояние без внешних библиотек. Для дашборда среднего размера это проще и легче в поддержке. `OnPush` +
signals дают предсказуемый и производительный change detection — важно для review UI с тяжёлыми изображениями.

**Следствие.** Никаких `@NgModule` для фич; всё через standalone-компоненты и `app.config.ts` провайдеры.
Lazy-loading через `loadComponent`/`loadChildren` на standalone-роутах где уместно.

## Состояние: signals, без NgRx (в MVP)

**Почему.** Объём состояния невелик (список билдов, текущий билд, очередь ревью, auth). Сервисы-хранилища с
`signal()`/`computed()`/`effect()` покрывают это без церемоний NgRx. NgRx оправдан при сложных
кросс-компонентных потоках и тайм-трэвеле — у нас этого нет. Это сознательное упрощение (анти-паттерн в
brainstorm: «тащит NgRx» → не надо).

**Паттерн.** `BuildStore`, `ReviewStore`, `AuthStore` — инжектируемые сервисы с приватными `signal` и
публичными `computed`/readonly-сигналами. Мутации через методы стора.

## UI: только Angular CDK, без тяжёлых китов

**Почему.** Angular Material целиком навязывает свою визуальную систему и вес; нам нужен функциональный,
не-генерик UI с акцентом на изображениях. CDK даёт нужные примитивы без визуального мнения: `overlay`
(модалки/попапы), `drag-drop` (шторка сравнения, маски), `scrolling` (virtual-scroll для списков превью),
`a11y` (focus-trap, live-announcer), `clipboard`, `portal`. Свои компоненты поверх CDK + дизайн через
frontend-design skill.

**Анти-паттерн.** Подключение Material целиком ради пары кнопок — запрещено спекой.

## image-compare: canvas vs CSS (B-10)

**Рекомендация: canvas для вьюпорта сравнения.** Три режима (side-by-side/onion/slider) с СИНХРОННЫМ
зумом/паном аккуратнее всего реализуются на `<canvas>`: одна матрица трансформации применяется к отрисовке
обеих картинок; onion — рисуем new поверх baseline с `globalAlpha`; slider — клиппинг по перетаскиваемой
границе. Canvas даёт пиксель-точный контроль и стабильную производительность при зуме.

**Когда CSS-трансформации.** Если canvas окажется избыточным — side-by-side можно сделать на `<img>` + CSS
`transform: scale/translate` с синхронизацией через общий сигнал трансформации. Но onion/slider и точный
зум всё равно тяготеют к canvas. Решение принять в начале Фазы 4 (обсудить в B-режиме), но синхронность
зума — нерушимое требование (F-26).

## Производительность review (F-27)

- **Thumbnails** для сетки превью (не полноразмерные PNG). Сервер может отдавать уменьшенные; иначе
  `loading="lazy"` + ограниченный размер отрисовки. Полноразмерные грузим только для открытого снапшота.
- **Virtual scroll** (CDK `cdk-virtual-scroll-viewport`) для длинных списков снапшотов/билдов.
- **Prefetch** следующего изображения в очереди ревью для плавной навигации хоткеями.
- **OnPush** + signals — отрисовка только при реальных изменениях.

## Типы и API-клиент

- Типы в `core/models/` ОБЯЗАНЫ зеркалить `specs/04-api-contract.md`. Держать их синхронно (можно
  генерировать из контракта/OpenAPI, если backend его отдаёт).
- Тонкий типизированный клиент: один метод на endpoint, без бизнес-логики в клиенте.
- Presigned URL картинок приходят с сервера — фронт просто вставляет в `<img>`/canvas, не ходит в MinIO напрямую.

## Аутентификация

- Auth на signals (`AuthStore`), HTTP-interceptor добавляет токен/использует cookie, route guard на
  защищённые роуты. GitLab OAuth — редирект-флоу (F-43), кнопка-вход.
- Критичное состояние НЕ в localStorage (только UI-преференции: тема, дефолтный режим сравнения).

## Дизайн-система и frontend-design skill

- Применять `frontend-design` skill (см. `skills/how-to-use-your-skills.md`) ДО написания UI-кода —
  особенно для review UI. Цель: профессиональный, отличимый от «генерик-AI» вид.
- Минимализм, акцент на изображениях; статусы — цвет + иконка + текст (не только цвет, ради a11y).
- Тёмная тема (F-52) удобна для разглядывания скриншотов, но не блокер MVP.

## Принцип

Дашборд — лицо инструмента. Список билдов может быть простым; review UI — нет. Синхронный зум, три режима,
хоткеи и плавность на сотнях изображений — это то, ради чего пользователь выберет Pixela вместо разглядывания
бинарных PNG в MR. Сюда вкладываем непропорционально много качества.
