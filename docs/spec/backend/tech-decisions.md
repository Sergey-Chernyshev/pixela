# Backend — Tech Decisions

> Обоснования выбора стека backend. Claude Code: следуй этим решениям; отклонения — через brainstorm с явным согласованием.

## NestJS

**Почему.** Структурированный, модульный фреймворк с DI «из коробки» — хорошо ложится на разделение по
feature-модулям (projects/builds/snapshots/diff/...). У Claude Code много опоры в обучающих данных по NestJS,
что снижает риск тонко-неверного кода. Декораторы + DTO + class-validator дают единообразную валидацию на
границе API. Один и тот же код легко запускать в двух режимах (http/worker) через bootstrap-переключатель.

**Альтернативы и почему нет.** Голый Express/Fastify — меньше структуры, больше ручного связывания для
проекта такого размера. Если критична максимальная производительность HTTP — можно поставить Fastify-адаптер
под NestJS (NestJS это поддерживает), не меняя архитектуру.

## Prisma

**Почему.** Типобезопасная схема как единый источник правды, понятные миграции, отличный DX. Схема в
`specs/03-data-model.md` — это контракт; Prisma делает её исполняемой и типизирует весь доступ к данным.
Content-addressable модель `Image` и уникальные ограничения снапшотов выражаются декларативно.

**Альтернативы.** TypeORM — мощнее в сложных случаях, но капризнее и менее предсказуем в миграциях.
Drizzle — легче и ближе к SQL, разумная альтернатива, но Prisma даёт более зрелый migration-флоу для команды.
Решение: Prisma; Drizzle — допустимая замена, если команда предпочитает SQL-first (обсудить в B-режиме).

## PostgreSQL

**Почему.** Надёжная реляционная БД; нам нужны связи (project→build→snapshot→baseline), транзакции (атомарный
пересчёт статуса билда при approve), индексы по частым запросам. JSON-поля при необходимости (ignore-маски).
Зрелый бэкап (`pg_dump`).

**Альтернативы.** MySQL — сопоставим, выбор по привычке команды. SQLite — заманчиво для self-host, но
параллельная запись из api+worker и рост истории делают Postgres надёжнее. Решение: Postgres.

## BullMQ (на Redis)

**Почему.** Diff — CPU-bound и должен быть асинхронным (инвариант #3). BullMQ даёт: очередь, ретраи с
backoff, конкурентность, переподхват незавершённых jobs после рестарта, изоляцию тяжёлой работы от API.
Redis нам всё равно полезен (сессии, счётчики шардов, дебаунс нотификаций).

**Альтернативы.** Свой воркер на pg-таблице-очереди — меньше зависимостей, но придётся переизобретать ретраи/
visibility timeout/конкурентность (риск тонких багов). Внешние брокеры (RabbitMQ/Kafka) — избыточны для
целевого масштаба. Решение: BullMQ.

## pixelmatch + sharp (diff)

**Почему.** pixelmatch — проверенный, простой попиксельный diff с порогом и anti-aliasing-обработкой; это
сознательно «дешёвая часть» (инвариант #5), не место для изобретений. sharp — быстрый декод/энкод PNG и
обработка (resize, нормализация, наложение масок) на libvips. Обе детерминированы при фиксированных версиях.

**Альтернативы/дополнения.** odiff — быстрее и layout/AA-aware на больших картинках; добавляем за фиче-флагом
(F-46), не как основу. looks-same — ещё вариант, не обязателен. Решение: pixelmatch+sharp как основа, odiff опционально.

## S3-совместимое (MinIO) для блобов

**Почему.** Блобы не место в БД (инвариант #2). MinIO даёт S3-API локально для self-host; код пишется под
стандартный S3-клиент и одинаково работает против MinIO и облачного S3. Content-addressable ключи (sha256) +
дедупликация дают экономию места/трафика.

**Альтернативы.** Файловая система — проще, но теряем presigned-URL, масштабирование и единый интерфейс;
переезд на объектное хранилище позже болезненнее. Решение: S3-совместимое (MinIO в self-host).

## Режим http/worker одним кодом

**Почему.** api и worker делят Prisma-клиент, доменные сервисы, конфиг. Один образ, переключение через
`API_MODE`. Меньше дублирования, проще деплой и тестирование.

## Версионирование и детерминизм

- Зафиксировать версии pixelmatch/sharp (точные, не `^`) и libvips через базовый образ — ради детерминизма diff.
- Node LTS зафиксирован (`.nvmrc`, `engines`).
- TypeScript strict; без `any` без обоснования.

## Принцип, объединяющий выбор

Стек выбран так, чтобы максимизировать опору Claude Code в обучающих данных и минимизировать класс задач, где
он генерирует правдоподобно-неверный код. Самая рискованная по корректности часть (baseline) вынесена в git
(см. `specs/06-baseline-strategy.md`), а не в серверную логику — это и есть главное архитектурное решение проекта.
