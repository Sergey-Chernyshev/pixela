# Agent 07 — Review UI (⭐ сердце продукта)

**Фаза:** 4 · **Фичи:** F-24..F-28 · **Зависит от:** Agent 06

## Задача
Реализовать review UI — экран сравнения одного снапшота с тремя режимами (side-by-side, onion-skin, slider),
синхронным зумом/паном и хоткеями. Это компонент, по которому судят весь инструмент. Здесь нельзя халтурить.

## Контекст (прочитать)
- `specs/08-frontend-spec.md` (Экран 3 — детально), `frontend/angular-decisions.md`, brainstorm B-10 (canvas vs CSS),
  `skills/how-to-use-your-skills.md` (frontend-design skill ОБЯЗАТЕЛЕН здесь).

## Что сделать
1. **Компонент `shared/image-compare`** с тремя режимами (F-25):
   - **side-by-side**: baseline | new (опц. третья панель diff).
   - **onion-skin**: new поверх baseline, слайдер прозрачности 0..100%.
   - **slider/шторка**: наложение с перетаскиваемой вертикальной границей.
   - Рекомендация B-10: реализовать через `<canvas>` для точного контроля (или обоснованно CSS).
2. **Синхронный зум и пан** (F-26): одна трансформация на обе картинки; зум колесом/кнопками, пан
   перетаскиванием; кнопки «1:1» и «fit». КРИТИЧНО: baseline и new двигаются строго синхронно.
3. **Корректная работа при разных размерах** baseline/new (рамки, не растягивать насильно).
4. **Review-панель**: метаданные (имя, browser, viewport, diffRatio, diffPixels); кнопки Approve/Reject
   (вызов API из Agent 08, но кнопки и состояние — здесь); индикатор «N/M отревьюено».
5. **Хоткеи** (F-28): `←/→` или `j/k` (prev/next changed), `a` (approve), `r` (reject), `1/2/3` (режим),
   `+/-` (зум). Видимая подсказка хоткеев.
6. **Производительность навигации** (F-27): prefetch следующего изображения в очереди; thumbnails в сетке
   (из Agent 06) — здесь грузим полноразмерные только для открытого снапшота.
7. **История** (F-33, UI-часть): лента прошлых approve по снапшоту (данные с сервера).
8. **Review queue**: после approve/reject — автопереход к следующему изменившемуся (F-51 как улучшение).

## Acceptance criteria
- [ ] Все три режима сравнения работают и переключаются (в т.ч. хоткеями 1/2/3).
- [ ] Зум и пан СИНХРОННЫ между baseline и new — это проверяется явно.
- [ ] Onion-slider прозрачности и шторка тащатся плавно.
- [ ] Разные размеры baseline/new отображаются корректно, без насильного растяжения.
- [ ] Хоткеи навигации/approve/reject/режима/зума работают; есть видимая подсказка.
- [ ] Переход между изменившимися снапшотами плавный (prefetch).
- [ ] Approve/Reject из UI вызывают API и обновляют состояние/прогресс.
- [ ] Применён frontend-design skill; компонент выглядит профессионально, не «генерик».

## Не делать
- Не грузить полноразмерные PNG для всей сетки превью (это убивает перформанс — только thumbnails).
- Не делать несинхронный зум (бесполезен для сравнения).
- Не обращаться к MinIO напрямую мимо presigned URL с сервера.
