# Agent 10 — Notifications

**Фаза:** 6 · **Фичи:** F-47, F-48 · **Зависит от:** Agent 04 (события билда), Agent 09 (GitLab каналы)

## Задача
Оркестратор уведомлений и каналы Telegram/Slack. На смену статуса билда — разослать сводку по включённым
каналам. Надёжно (падение нотификации не ломает билд) и без спама (дебаунс по статусу).

## Контекст (прочитать)
- `specs/09-integrations.md` (Telegram/Slack, архитектура notifications-модуля).

## Что сделать
1. **Оркестратор** `notifications.service.ts`: подписан на доменные события (build.statusChanged →
   REVIEW_REQUIRED / PASSED / ERROR). Определяет включённые каналы per-project и рассылает.
2. **Общий интерфейс** `NotificationChannel { notify(event): Promise<void> }`. Каналы:
   - `telegram.channel.ts` (F-47): Bot API `sendMessage`; конфиг `TELEGRAM_BOT_TOKEN` + chatId per-project.
   - `slack.channel.ts` (F-48): Incoming Webhook; `SLACK_WEBHOOK_URL` per-project.
   - (GitLab каналы из Agent 09 подключаются через тот же оркестратор.)
3. **Конфиг каналов** per-project (`notification-config`): какие включены, их параметры.
4. **Дебаунс/идемпотентность**: одно сообщение на смену статуса билда, НЕ на каждый снапшот.
5. **Надёжность**: fire-and-forget, ошибки логируются (без секретов), не влияют на обработку билда.
6. Формат сообщений по `09-integrations.md` (проект, ветка, counts, ссылка на билд).

## Acceptance criteria
- [ ] При REVIEW_REQUIRED/PASSED приходит сообщение в настроенный Telegram-чат со сводкой и ссылкой.
- [ ] При REVIEW_REQUIRED/PASSED приходит сообщение в Slack (если включён).
- [ ] Каналы включаются/настраиваются per-project; выключенные не шлют.
- [ ] Одно сообщение на смену статуса (нет спама по снапшотам).
- [ ] Падение/таймаут канала логируется и НЕ ломает обработку билда.
- [ ] Тесты (с моками HTTP): маппинг события в сообщение, выбор включённых каналов, изоляция ошибок.
