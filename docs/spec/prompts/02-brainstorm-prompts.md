## Глобальные развилки (обсудить на старте)

**B-01. Монорепо или отдельные репозитории?**
api + web + sdk в одном монорепо (проще атомарные изменения контракта) против отдельных репо (чище
границы). Рекомендация спеки: монорепо (npm/pnpm workspaces). Обсуди, если есть причины иначе.

**B-02. Baseline режим A или B?** (см. `06-baseline-strategy.md`)
A: Playwright владеет baseline-файлами, Pixela — слой ревью. B: Pixela хранит baseline в БД + синк с git.
Рекомендация: начать с A (проще, не дублирует источник правды). Когда выбрать B?

**B-03. Сессии: серверные (Redis) или JWT?**
Для self-host малой команды серверные проще отзывать. JWT — stateless. Реши и зафиксируй.

## Фаза 1 (ingestion/storage)

**B-04. Двухшаговая заливка vs multipart за раз?**
Спека предлагает двухшаговый протокол (объявить sha → залить если нужно) ради дедупликации БЕЗ заливки
байтов. Альтернатива: всегда multipart, дедуп на сервере после приёма (проще клиент, больше трафика).
Трейд-офф: трафик CI vs сложность протокола. Рекомендация: двухшаговый (CI-трафик дороже).

**B-05. Thumbnails: генерировать при ingestion или on-demand?**
Превью для сетки. При ingestion (быстрее отдача, больше места/работы на приёме) vs on-demand с кэшем
(лениво, первый показ медленнее). Реши с учётом F-27.

## Фаза 2 (diff)

**B-06. Разные размеры baseline/new — как обрабатывать?**
MVP-рекомендация: разные размеры = CHANGED (diffRatio=1), side-by-side, без ресайза. Альтернатива:
padding до общего холста + diff. «Умный» ресайз — НЕ делать (искажает чувствительность). Подтверди выбор.

**B-07. diffRatioThreshold по умолчанию: 0 или малое значение?**
0 = любой изменённый пиксель → CHANGED (строго, но возможен шум). Малое (напр. 0.001) = терпит крошечный
шум. Рекомендация: 0 по умолчанию + per-project override; полагаться на pixelThreshold для anti-aliasing.

## Фаза 3 (Playwright)

**B-08. Diff считать на клиенте (Playwright) или только на сервере?**
В режиме A Playwright уже умеет diff локально. Считать ли его источником, а Pixela только для ревью? Или
всегда пересчитывать на сервере ради единообразия UI/порогов? Рекомендация: сервер — источник статуса
(единые пороги), клиентский diff — опционально для быстрого фейла в CI.

**B-09. Шардинг: как надёжно понять, что все шарды пришли?**
Счётчик в Redis по (buildId) до parallelTotal? Таймаут на случай упавшего шарда? Обсуди idempotent
финализацию и поведение при недоехавшем шарде.

## Фаза 4 (frontend)

**B-10. image-compare: canvas или CSS-трансформации?**
Canvas — точный контроль зума/наложения/шторки, сложнее. CSS transform — проще, но синхронный зум двух
картинок и onion/slider аккуратнее на canvas. Рекомендация: canvas для compare-вьюпорта. Обсуди.

**B-11. Live-обновление: polling или SSE?** (F-44)
Polling проще, SSE приятнее во время COMPARING. Рекомендация: начать с polling, SSE как улучшение.

## Фаза 5 (approve/git)

**B-12. Git-native approve: CLI, commit через API, или MR?** (см. `06`, Approve→Git)
MVP-минимум — CLI `pixela pull-baseline`. Commit через GitLab API — удобнее, нужен write-токен. Отдельный
MR — безопаснее, больше частей. Какой уровень автоматизации для v1? Рекомендация: CLI гарантированно +
commit-через-API если есть токен.

**B-13. Блокировать ли merge при REVIEW_REQUIRED?**
Commit status `failed` блокирует (строго) vs `pending`/`success` (не блокирует). Это политика команды.
Сделать конфигурируемым per-project.
