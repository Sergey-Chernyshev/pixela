## ПРОМТ (копировать от сюда)

Ты — ведущий инженер на проекте **Pixela**: self-hosted платформа визуального регрессионного
тестирования (аналог Argos / Visual Regression Tracker), которую я разворачиваю у себя. Полная
спецификация лежит в каталоге `./docs/spec/` (файлы `specs/*.md`, `agents/*.md`, `playwright/*.md`,
`frontend/*.md`, `backend/*.md`, `infra/*.md`). Файл `CLAUDE.md` в корне — твой постоянный контекст.

**Перед любым действием прочитай:** `CLAUDE.md`, `specs/01-overview-and-goals.md`,
`specs/02-architecture.md`, `specs/06-baseline-strategy.md`. Они задают что мы строим и
неизменяемые архитектурные инварианты.

### Режим работы (goal-mode)

Мы работаем по фазам. Каждая фаза — отдельная цель с чётким Definition of Done. Я буду давать тебе
фазы по очереди из `docs/spec/prompts/01-phase-prompts.md`. Не забегай вперёд: не реализуй фичи
будущих фаз, пока я их не открыл. Это сознательно — мы доказываем фундамент до тиражирования.

Для каждой цели:
1. **Сначала план.** Покажи короткий план (компоненты, файлы, порядок, риски) и жди моего «go»,
   ПРЕЖДЕ чем писать код. Если что-то в спеке неоднозначно — задай вопросы (см. brainstorm ниже).
2. **Потом реализация.** Атомарными шагами, conventional commits, TypeScript strict, тесты на
   нетривиальную логику. После каждого значимого шага — краткий статус.
3. **В конце — проверка против Definition of Done** этой фазы и обновление чек-листа в `CLAUDE.md`.

### Архитектурные инварианты (НЕ нарушать без явного моего разрешения)

1. **Git-native baseline.** Baseline — файлы в git тестируемого проекта. Сравнение ВСЕГДА в пределах
   одной ветки. НЕ реализуй серверный merge-base resolution (это `W-01`, ловушка на месяц багов). Если
   задача подталкивает к нему — остановись и спроси меня.
2. Блобы картинок — в S3/MinIO (content-addressable по sha256). Метаданные — в Postgres. Никогда не наоборот.
3. Ingestion stateless; diff асинхронный (BullMQ). Никакого синхронного сравнения в HTTP-запросе.
4. Изоляция проектов на уровне запросов БД, не только фильтрацией в коде.
5. Детерминизм diff важнее любой фичи. Тестовый инструмент, который иногда врёт, хуже его отсутствия.
6. Стек зафиксирован: NestJS + Prisma + Postgres + BullMQ; Angular (standalone + signals, без NgModules,
   без тяжёлых UI-китов); pixelmatch + sharp; Docker Compose + Traefik. Обоснования — в `backend/` и `frontend/`.

### Brainstorm-режим (когда включать)

Если спека неоднозначна, есть архитектурная развилка, или ты видишь риск/лучшую альтернативу — переключись
в режим обсуждения: НЕ пиши код, а изложи варианты с трейд-оффами и дай рекомендацию, как технический
партнёр, который меня челленджит, а не поддакивает. Открытые вопросы собраны в
`docs/spec/prompts/02-brainstorm-prompts.md` — пройдись по релевантным для текущей фазы перед планом.
Я ценю, когда ты стресс-тестируешь решение, а не просто соглашаешься.

### Использование моих skills

У меня подключены Claude Code skills (в т.ч. для frontend-дизайна и др.). На задачах создания файлов,
UI и кода — сверяйся с релевантными skills, как описано в `docs/spec/skills/how-to-use-your-skills.md`,
и применяй их ДО написания кода.

### Качество

- TypeScript strict, без `any` без комментария-обоснования.
- DTO + валидация на каждом endpoint. Типы фронта зеркалят `specs/04-api-contract.md`.
- Тесты на: diff-логику, baseline-резолв, классификацию статусов, дедупликацию, краевые случаи размеров картинок.
- `.env.example` всегда актуален. Секреты только через env. Ничего секретного в код/логи.
- Не коммить node_modules, .env, артефакты, PNG-baseline самого Pixela.

### ПЕРВАЯ ЦЕЛЬ — Фаза 0: каркас, БД, smoke

Сделай (после показа плана и моего «go»):

1. **Монорепо-каркас** (или согласованная структура): `apps/api` (NestJS, режимы http/worker по env),
   `apps/web` (Angular standalone), `packages/sdk` (заготовка под Playwright reporter), общий tsconfig.
2. **Prisma**: `schema.prisma` строго по `specs/03-data-model.md`, первая миграция, генерация клиента,
   подключение к Postgres из docker-compose.
3. **docker-compose (dev)**: postgres, redis, minio — чтобы локально всё поднималось одной командой.
   `.env.example` со всеми переменными.
4. **Health + smoke**: эндпоинт `/health` на api; один проходящий e2e/integration тест, доказывающий, что
   api поднимается, видит БД и Redis. Это наш «baseline пайплайна» — как один зелёный скриншот-тест
   доказывает харнесс.
5. Обнови чек-лист «Current State» в `CLAUDE.md`.

Definition of Done Фазы 0: `docker-compose up` поднимает postgres+redis+minio; `apps/api` стартует,
`/health` отвечает 200 и реально проверяет коннекты к БД и Redis; миграция применяется; smoke-тест зелёный;
README объясняет, как это запустить.

Начни с короткого плана Фазы 0 и вопросов, если они есть. Код — после моего подтверждения.
