# How To Use Your Claude Code Skills

> Как подключить твои Claude Code skills к проекту Pixela, чтобы агенты справились максимально круто.
> Skills — это переиспользуемые наборы инструкций/инструментов, которые Claude Code читает ДО работы над задачей.

## Принцип

Перед созданием файлов, кода или UI Claude Code должен сверяться с релевантными skills и применять их.
Для Pixela самые полезные точки приложения — frontend (review UI), создание документов/конфигов, и любые
твои собственные skills под ваш стек/конвенции.

## Frontend-design skill (критично для Фазы 4)

Review UI — лицо инструмента. Здесь skill по фронтенд-дизайну даёт непропорциональный выигрыш: помогает
сделать профессиональный, отличимый от «генерик-AI» интерфейс с грамотными токенами, типографикой и
компоновкой.

**Когда применять:** Agent 06 (shell) и особенно Agent 07 (review UI) — ПЕРЕД написанием компонентов.

**Как (в промте фазы):**
> «Перед реализацией UI сверься с frontend-design skill и применяй его рекомендации по дизайн-токенам,
> типографике, цветовой системе и компоновке. Review UI (image-compare, 3 режима, зум) должен выглядеть
> профессионально. Покажи, какие решения skill'а ты применяешь.»

**На что обратить внимание для Pixela:**
- акцент на изображениях, минималистичный «хром» интерфейса;
- статусы билдов/снапшотов: цвет + иконка + текст (не только цвет — a11y);
- плотная, но не перегруженная сетка превью; явная иерархия в review-панели;
- хоткеи с видимой подсказкой;
- тёмная тема как опция (удобно разглядывать скриншоты).

## Skills для создания файлов/документов

Если у тебя есть skills для генерации конфигов, README, или структурированных документов — они полезны на:
- Agent 00 (scaffold: tsconfig, eslint, docker-compose, .env.example, README);
- Agent 11 (deploy: prod compose, backup-скрипт, документация восстановления).

Применять до написания соответствующих артефактов.

## Твои собственные / командные skills

Если у тебя есть skills, кодирующие ваши конвенции (стиль кода, структура NestJS-модулей, Angular-паттерны,
правила коммитов, CI-шаблоны) — подключи их в начале и сошлись на них в master-промте. Это выровняет
сгенерированный код с тем, как пишет твоя команда, и снизит переделки.

**Как сослаться (в master-промте):**
> «У меня подключены skills с нашими конвенциями: [перечисли]. Следуй им для структуры модулей, стиля кода и
> коммитов. При конфликте между skill и спекой — спроси меня, какой приоритет.»

## Порядок интеграции skills в работу

1. **В начале сессии** (вместе с master-промтом из `prompts/00-goal-mode-master-prompt.md`): перечисли
   доступные skills и попроси Claude Code учитывать их.
2. **Перед каждой фазой**: в фазовом промте напомни про релевантный skill (особенно frontend-design на Фазе 4).
3. **На задачах создания файлов/кода**: Claude Code сверяется со skill ДО написания, не после.
4. **При конфликте** skill ↔ спека ↔ инвариант: приоритет у архитектурных инвариантов (`CLAUDE.md`), затем
   спека, затем skill; в спорных случаях — спросить тебя (brainstorm-режим).

## Чего НЕ делать со skills

- Не давать skill переопределять архитектурные инварианты (git-native baseline, блобы в S3, async diff,
  изоляция проектов, детерминизм, зафиксированный стек). Инварианты — выше skills.
- Не подключать skills, противоречащие зафиксированному стеку (напр. навязывающие React/Vue вместо Angular,
  или Material целиком) — это создаст шум.

## Сводка: где какой skill даёт максимум

| Фаза / Agent | Skill | Зачем |
|--------------|-------|-------|
| Фаза 0 / Agent 00, 11 | создание файлов/доков | каркас, конфиги, deploy-артефакты, README |
| Фаза 4 / Agent 06, 07 | **frontend-design** | профессиональный review UI — лицо продукта |
| Все фазы | ваши конвенции (если есть) | код в стиле команды, меньше переделок |

> Главное: skills ускоряют и поднимают качество, но архитектурные инварианты из `CLAUDE.md` и решения из
> `specs/` — это рамка, внутри которой skills работают, а не наоборот.
