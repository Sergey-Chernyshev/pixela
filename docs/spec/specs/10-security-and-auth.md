# 10 — Security & Auth

## Два контура аутентификации

| Контур | Кто | Механизм | Доступ |
|--------|-----|----------|--------|
| Ingestion | CI / Playwright SDK | `Authorization: ApiKey <key>` | только приём в свой проект |
| Dashboard | люди | сессия/JWT (login или GitLab OAuth) | чтение + approve по membership |

## API-ключи (F-34)

- Генерация: криптослучайный ключ (напр. `pxl_` + 32 байта base62). Показывается **один раз** при создании.
- Хранение: только `keyHash` (bcrypt/argon2 или SHA-256 с солью — для API-ключей подходит быстрый хэш с солью,
  т.к. ключ высокоэнтропийный; bcrypt избыточен, но не вреден). Сырой ключ в БД НЕ хранится.
- Проверка: guard хэширует входящий ключ, ищет по `keyHash`, определяет `projectId`.
- Отзыв: удаление строки `ApiKey`. `lastUsedAt` обновляется для аудита.
- Один проект может иметь несколько ключей (ротация без даунтайма).

## Сессии дашборда

- Пароли пользователей: argon2id (или bcrypt). Никогда в открытом виде.
- JWT (короткоживущий) + refresh, ИЛИ серверная сессия в Redis. Выбор: серверная сессия в Redis проще
  отзывать (logout, бан). JWT — если нужен stateless. Для self-host малой команды — серверная сессия ок.
- Cookie: `HttpOnly`, `Secure`, `SameSite=Lax`.

## Изоляция проектов (F-36)

- Каждый ресурс (build, snapshot, baseline) принадлежит проекту.
- Guard на dashboard-endpoints проверяет membership: `userId` ∈ members(projectId ресурса).
- Ingestion-guard ограничивает запись строго в проект API-ключа (нельзя залить в чужой проект).
- В ответах API — никаких данных проектов, к которым нет доступа. Проверять на уровне query (where projectId in user's projects), не только на уровне фильтрации в коде.

## Защита загрузки блобов

- Проверка sha256: загруженные байты хэшируются на сервере и сверяются с заявленным `imageSha256`.
  Несовпадение → 400 `SNAPSHOT_HASH_MISMATCH`. (Защита от подмены/коррупции.)
- Лимит размера PNG (`IMAGE_TOO_LARGE`).
- Проверка Content-Type и магических байтов PNG (не доверять расширению).
- Presigned URL на чтение — короткоживущие (1 ч), привязаны к конкретному объекту.

## Секреты

- Всё через env: `DATABASE_URL`, `REDIS_URL`, `S3_*`, `GITLAB_TOKEN`, `TELEGRAM_BOT_TOKEN`, `SLACK_WEBHOOK_URL`,
  `JWT_SECRET`/`SESSION_SECRET`, `GITLAB_OAUTH_*`.
- `.env.example` всегда актуален и закоммичен; реальный `.env` — никогда.
- Секреты не логируются. Маскировать токены в логах.

## Rate limiting

- Ingestion: лимит запросов на проект (защита от шального CI-цикла, заливающего тысячи скриншотов).
- Auth login: лимит попыток на IP/email (защита от перебора).

## Reverse-proxy / TLS

- Traefik (или nginx) терминирует TLS. Pixela-сервисы слушают только внутри docker-сети.
- MinIO-консоль (9001) — НЕ публиковать наружу или закрыть basic-auth/VPN.
- CORS на API: разрешить только origin дашборда.

## Аудит

- `ApprovalEvent` — кто/когда/что принял (встроенный аудит ревью).
- `ApiKey.lastUsedAt` — последняя активность ключа.
- Логировать (без секретов): создание/отзыв ключей, логины, approve/reject.

## Что НЕ делаем в v1 (явно)

- Сложный RBAC (только OWNER/MEMBER).
- SAML/SCIM (`W-07`).
- Шифрование блобов at-rest на уровне приложения (полагаемся на шифрование диска/хранилища инфраструктуры).
