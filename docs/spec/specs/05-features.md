# 05 — Features (полный список с приоритетами)

> Приоритизация по **MoSCoW**: Must / Should / Could / Won't (в v1).
> Каждая фича помечена фазой постройки (см. README). Это чек-лист охвата для Claude Code.

## MUST (без этого продукт не существует)

### Приём данных
- **F-01** Создание билда с метаданными (branch, commitSha, ciBuildId, mrIid). [Фаза 1]
- **F-02** Двухшаговая заливка скриншота с дедупликацией по sha256 (CAS). [Фаза 1]
- **F-03** Проверка целостности: байты загруженного блоба совпадают с заявленным хэшем. [Фаза 1]
- **F-04** Финализация билда; вычисление `REMOVED` снапшотов. [Фаза 1]
- **F-05** Идемпотентная заливка (ретрай CI не плодит дубли — upsert по уникальному ключу). [Фаза 1]

### Хранение
- **F-06** S3/MinIO storage-слой с presigned URL на чтение. [Фаза 1]
- **F-07** Content-addressable: один PNG = один блоб по sha256, дедупликация. [Фаза 1]
- **F-08** Метаданные в Postgres, блобы в S3 (строгое разделение). [Фаза 1]

### Сравнение
- **F-09** Async diff через очередь (BullMQ): API enqueue, worker обрабатывает. [Фаза 2]
- **F-10** pixelmatch-сравнение → diffRatio, diffPixels, генерация diff-PNG. [Фаза 2]
- **F-11** Порог чувствительности (threshold) — per-project и/или per-snapshot override. [Фаза 2]
- **F-12** Классификация: UNCHANGED / CHANGED / NEW / REMOVED. [Фаза 2]
- **F-13** Агрегатный статус билда: PASSED / REVIEW_REQUIRED / ERROR. [Фаза 2]
- **F-14** Ретрай упавших diff-jobs, изоляция ошибок (один битый скриншот не валит билд). [Фаза 2]

### Playwright-интеграция
- **F-15** `@pixela/playwright-reporter` — кастомный reporter, заливающий скриншоты в Pixela. [Фаза 3]
- **F-16** Конфиг через `pixela.config` / env: apiUrl, projectKey, branch, commit, ciBuildId. [Фаза 3]
- **F-17** Автоопределение branch/commit/ciBuildId из GitLab CI env (CI_COMMIT_*, CI_PIPELINE_*). [Фаза 3]
- **F-18** Поддержка шардинга Playwright (агрегация скриншотов из нескольких шардов в один билд). [Фаза 3]
- **F-19** Soft mode: не валить тест-прогон при визуальном различии (различие ревьюится в дашборде). [Фаза 3]

### Дашборд: просмотр
- **F-20** Список проектов пользователя. [Фаза 4]
- **F-21** Лента билдов проекта с фильтрами (branch, status) и пагинацией. [Фаза 4]
- **F-22** Детали билда: список снапшотов, счётчики (unchanged/changed/new/removed). [Фаза 4]
- **F-23** Быстрый фильтр «показать только изменившиеся» (changed/new). [Фаза 4]

### Дашборд: review UI (самое важное по UX)
- **F-24** Просмотр одного снапшота: baseline / new / diff. [Фаза 4]
- **F-25** **3 режима сравнения:** side-by-side, onion-skin (наложение с прозрачностью), slider (шторка). [Фаза 4]
- **F-26** Синхронный зум и панорамирование по baseline и new одновременно. [Фаза 4]
- **F-27** Ленивая загрузка превью списка (сотни PNG не должны вешать UI). [Фаза 4]
- **F-28** Навигация клавиатурой между изменившимися снапшотами (next/prev, approve/reject хоткеи). [Фаза 4]

### Approve workflow
- **F-29** Approve/Reject одного снапшота → обновление Baseline + ApprovalEvent. [Фаза 5]
- **F-30** Пакетный approve-all / reject-all для билда. [Фаза 5]
- **F-31** Пересчёт статуса билда после approve (не осталось нерешённых → PASSED). [Фаза 5]
- **F-32** **Git-native действие:** approve готовит коммит/MR с обновлёнными baseline-PNG в репо проекта. [Фаза 5]
- **F-33** История: лента кто/когда/что принимал по снапшоту. [Фаза 5]

### Аутентификация и изоляция
- **F-34** API-ключи проекта (хэшируются, показываются раз). [Фаза 1]
- **F-35** Логин дашборда (email+пароль). [Фаза 4]
- **F-36** Изоляция проектов: пользователь видит только свои (по membership). [Фаза 4]

### Инфраструктура
- **F-37** docker-compose: api, worker, web, postgres, redis, minio, traefik. [Фаза 7]
- **F-38** Миграции Prisma, `.env.example`, healthchecks. [Фаза 0/7]
- **F-39** Reverse-proxy + TLS. [Фаза 7]
- **F-40** Бэкап БД и стратегия восстановления. [Фаза 7]

## SHOULD (сильно желательно, но продукт жив и без)

- **F-41** Статус билда в GitLab MR: pipeline/commit status через GitLab API. [Фаза 5]
- **F-42** Комментарий в GitLab MR со сводкой визуальных изменений + ссылкой на билд. [Фаза 5]
- **F-43** GitLab OAuth для входа в дашборд. [Фаза 4/6]
- **F-44** SSE live-обновление статуса билда во время COMPARING. [Фаза 4]
- **F-45** Ignore-области (маски) на уровне снапшота, задаваемые в дашборде. [Фаза 6]
- **F-46** odiff как альтернативный движок (layout-aware, anti-aliasing). [Фаза 2/6]
- **F-47** Уведомления в Telegram (билд требует ревью / билд passed). [Фаза 6]
- **F-48** Уведомления в Slack. [Фаза 6]
- **F-49** Роли OWNER/MEMBER в проекте. [Фаза 4]

## COULD (приятно, если останется время)

- **F-50** Тренд-график diffRatio по истории для одного скриншота.
- **F-51** Bulk-операции с клавиатуры в режиме «review queue» (как в Argos).
- **F-52** Тёмная тема дашборда.
- **F-53** Поиск снапшота по имени внутри билда.
- **F-54** Метрика «flaky»: скриншот, который то changed, то unchanged без изменений кода.
- **F-55** Экспорт diff-отчёта билда как статичного HTML (артефакт CI).
- **F-56** Webhook-исходящие события (build.completed) для интеграций.

## WON'T (явно НЕ в v1 — чтобы Claude Code не уходил в это)

- **W-01** Серверный baseline resolution через git merge-base. (Архитектурный инвариант.)
- **W-02** Visual AI / ML «значимость изменения».
- **W-03** Облачный кросс-браузерный грид исполнения тестов.
- **W-04** Cypress / Storybook интеграции.
- **W-05** Мультитенантный биллинг, организации, сложный RBAC.
- **W-06** Мобильные нативные скриншоты.
- **W-07** SAML/SCIM enterprise SSO.
