# 08 — Frontend Spec (Angular)

> Дашборд — самая дорогая часть по UX. Пользователь судит весь инструмент по качеству review UI.
> Стек: Angular (standalone components, signals, OnPush), Angular CDK. Без тяжёлых UI-китов.

## Архитектура приложения

```
src/app/
├── app.config.ts            # standalone bootstrap, провайдеры, роутинг
├── app.routes.ts            # роуты (lazy где уместно)
├── core/
│   ├── api/                 # типизированный HTTP-клиент (под 04-api-contract.md)
│   ├── auth/                # auth-сервис, guard, interceptor (Bearer/cookie)
│   └── models/              # TS-типы, ЗЕРКАЛЯТ API-контракт
├── features/
│   ├── projects/            # список проектов
│   ├── builds/              # лента билдов, детали билда
│   ├── review/              # ⭐ review UI (главная ценность)
│   └── settings/            # проект, API-ключи, участники
└── shared/
    ├── ui/                  # переиспользуемые компоненты (кнопки, статус-бейджи)
    └── image-compare/       # ⭐ компонент сравнения изображений (3 режима)
```

## Состояние

- **Signals** для локального и shared-состояния. Без NgRx в MVP (избыточно).
- Сервисы-хранилища с `signal()` / `computed()` (напр. `BuildStore`, `ReviewStore`).
- `OnPush` change detection везде.
- Polling статуса билда через `interval` + refetch, ИЛИ SSE (`EventSource`) на `/builds/:id/events` (F-44).
- НЕ хранить критичное состояние в localStorage (только UI-преференции: тема, режим сравнения по умолчанию).

## Экран 1: Список билдов (F-21, F-22)

- Таблица/карточки билдов: branch, commit (short), статус-бейдж, счётчики (✓480 ✎12 +3 −1), время, ссылка на CI-джобу.
- Фильтры: branch (dropdown), status (multi), поиск по commit.
- Пагинация (server-side, `?page=`).
- Клик по билду → детали.
- Статус-бейджи цветом: PASSED (зелёный), REVIEW_REQUIRED (жёлтый), REJECTED (красный), COMPARING (синий, спиннер), ERROR (серый/красный).

## Экран 2: Детали билда (F-22, F-23)

- Заголовок: branch, commit, статус, кнопки **Approve all** / **Reject all** (если REVIEW_REQUIRED).
- Тоггл «только изменившиеся» (default ON) — скрывает UNCHANGED, оставляет CHANGED/NEW/REMOVED.
- Сетка превью снапшотов (ленивая загрузка, F-27): миниатюра + имя + бейдж статуса + diffRatio.
- Клик по снапшоту → review (Экран 3).
- Группировка/сортировка: по статусу (changed сверху), по имени, по diffRatio (убыв.).

## Экран 3: Review UI (⭐ F-24..F-28 — сердце продукта)

### Компонент `image-compare` — 3 режима (F-25)

```
Режим 1: Side-by-side
  [ baseline ] | [ new ]      (+ опц. третья панель diff)
  Синхронный зум/пан обеих панелей.

Режим 2: Onion-skin (наложение)
  new поверх baseline с регулируемой прозрачностью (слайдер 0..100%).
  Видно «проявление» изменения.

Режим 3: Slider (шторка)
  baseline и new наложены, вертикальная шторка делит экран,
  тащишь её влево/вправо мышью. Классика BackstopJS.
```

### Требования к `image-compare`

- **Синхронный зум и панорамирование** (F-26): зум колесом/кнопками, пан перетаскиванием; baseline и new
  двигаются СИНХРОННО (одна трансформация на обе). Это критично — несинхронный зум бесполезен для сравнения.
- Кнопка «1:1» (реальные пиксели) и «fit» (вписать).
- Подсветка diff-области (из diff-PNG) опциональным оверлеем.
- Корректная работа при разных размерах baseline/new (показать рамки, не растягивать насильно).

### Review-панель

- Метаданные: имя, browser, viewport, diffRatio, diffPixels.
- Кнопки **Approve** / **Reject** (большие, очевидные).
- **Хоткеи** (F-28): `←/→` или `j/k` — prev/next changed; `a` — approve; `r` — reject; `1/2/3` — режим сравнения; `+/-` — зум.
- Индикатор прогресса: «3 / 12 изменений отревьюено».
- После approve/reject — автопереход к следующему изменившемуся (review queue, F-51 как улучшение).
- История снизу/сбоку (F-33): лента прошлых approve по этому скриншоту (кто, когда, в каком билде).

### Производительность review (F-27)

- Превью в списке — отдельные уменьшенные изображения (сервер может отдавать thumbnail, либо CSS-масштаб
  с lazy `loading="lazy"`). Не грузить полноразмерные PNG для сетки превью.
- Полноразмерные baseline/new/diff грузятся только при открытии конкретного снапшота.
- Предзагрузка следующего изображения в очереди (prefetch) для плавной навигации хоткеями.
- Виртуализация длинного списка превью (Angular CDK `cdk-virtual-scroll`).

## Аутентификация (F-35, F-36, F-43)

- Login-экран (email+пароль). JWT/cookie. HTTP-interceptor добавляет токен.
- Guard на все роуты кроме login.
- GitLab OAuth-кнопка (F-43, Should): редирект на `/api/v1/auth/gitlab`.
- Пользователь видит только свои проекты (сервер фильтрует по membership; фронт не показывает чужое).

## Дизайн-система

- Минималистичная, функциональная. Акцент на изображениях, не на хроме интерфейса.
- Используй `frontend-design` skill (см. `skills/how-to-use-your-skills.md`) для качественного, не-генерик UI.
- Тёмная тема (F-52, Could) — удобно для разглядывания скриншотов, но не блокер.
- Никакого Angular Material целиком — только CDK (overlay, drag, virtual-scroll, a11y) + свои компоненты.

## Технические заметки для Claude Code

- Типы в `core/models/` ОБЯЗАНЫ соответствовать `specs/04-api-contract.md`. Генерируй их из контракта.
- API-клиент — тонкий, типизированный, один метод на endpoint.
- Презайн-URL картинок приходят с сервера; фронт просто вставляет в `<img>`/canvas. Не обращаться к MinIO напрямую с фронта мимо presigned.
- Для slider/onion/zoom удобнее `<canvas>` или CSS-трансформации; canvas даёт точный контроль зума/наложения. Выбор на усмотрение, но синхронность зума — обязательна.
