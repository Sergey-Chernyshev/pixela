# 01 — Overview & Goals

## Цель проекта

Построить self-hosted платформу визуального регрессионного тестирования (**Pixela**), которую можно
развернуть на собственном сервере и использовать как замену облачным Argos / Percy / Chromatic, с
интеграцией в Playwright-тесты и GitLab CI.

## Проблема, которую решаем

Голый Playwright (`toHaveScreenshot()`) уже умеет: снимать скриншоты, делать pixel-diff, хранить
baseline как файлы в git. Чего у него нет и что даёт боль:

1. **Нет приятного ревью diff.** Бинарный PNG в GitLab MR смотреть невозможно — не видно, что именно изменилось.
2. **Нет истории по CI.** Невозможно посмотреть «как менялся этот скриншот за последние 20 прогонов».
3. **Нет управляемого approve-флоу.** `--update-snapshots` обновляет всё подряд локально, без выборочного апрува и без следа кто/когда/что принял.
4. **Нет статуса в MR.** Нельзя заблокировать merge, пока визуальные изменения не отревьюены.
5. **Нет агрегации по шардам/джобам.** Параллельные прогоны не собираются в один просматриваемый билд.

Pixela закрывает ровно эти пять разрывов, не пытаясь переизобрести то, что Playwright уже делает хорошо.

## Что НЕ входит в scope (явные границы)

- **Не** собственный движок снятия скриншотов — снимает Playwright, мы принимаем готовые PNG.
- **Не** серверный baseline resolution через git merge-base (см. `06-baseline-strategy.md`).
- **Не** кросс-браузерный грид/облако исполнения тестов — тесты бегут в твоём CI.
- **Не** Visual AI / ML-определение «значимости» изменения в v1 (можно как пост-MVP эксперимент).
- **Не** поддержка Cypress/Storybook в v1 (архитектура не должна мешать добавить позже, но не делаем сейчас).
- **Не** биллинг, мультитенантные организации, RBAC сложнее «owner/member» в v1.

## Критерии успеха v1 (Definition of Done для всего продукта)

Pixela считается готовым к использованию, когда выполняется сквозной сценарий:

1. Тестовый проект на Playwright подключает `@pixela/playwright-reporter`, прогоняет тесты в GitLab CI.
2. Скриншоты с метаданными (имя, браузер, viewport, ветка, commit, build id) приезжают в Pixela.
3. Pixela асинхронно сравнивает каждый скриншот с baseline, генерирует diff-изображения.
4. В дашборде виден билд со списком: unchanged / changed / new / removed; changed подсвечены.
5. Ревьюер открывает изменившийся скриншот, видит 3 режима сравнения (side-by-side / onion / slider),
   синхронный зум, и жмёт Approve или Reject — поштучно или пачкой.
6. Approve готовит коммит/MR с обновлёнными baseline-снапшотами в репозиторий тестируемого проекта.
7. Статус билда (passed / changes requested) отражается в GitLab MR как pipeline status / комментарий.
8. Видна история: для каждого скриншота — лента прошлых версий и кто что принимал.
9. Всё развёрнуто через `docker-compose up` за reverse-proxy с TLS, переживает рестарт, есть бэкап БД.

## Не-цели по качеству (сознательные упрощения v1)

- Перформанс: цель — комфортно держать билды до ~500 скриншотов и историю на ~50 проектов. Не гипермасштаб.
- Аутентификация дашборда: локальные аккаунты или GitLab OAuth. Не SAML/SCIM.
- Диалект CI: GitLab first. GitHub/прочее — через generic REST, но не вылизываем.

## Персона пользователя

Команда из ~7–8 человек, frontend на Angular, тесты на Playwright, CI в GitLab, хочет
self-host без облачных зависимостей и без оплаты за скриншоты. Ревью визуальных изменений — часть
code review в MR.
